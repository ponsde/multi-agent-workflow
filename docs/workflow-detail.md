## 讨论机制详解

### 为什么需要讨论？

Debug 修了代码，但 Leader 不认同修改方式——这很正常，不同模型有不同判断。这时候需要一个结构化的讨论机制，而不是无限来回。

### 轮次计算

```
1 轮 = Debug→Leader + Leader→Debug

第 1 轮：Debug 修复 → Leader 审查不满意 → Leader 改代码发回 Debug
第 2 轮：Debug 审查 → Leader 审查不满意 → Leader 改代码发回 Debug
第 3 轮：Debug 审查 → Leader 审查…
         └── 不满意？→ 停止！上报用户
         └── 满意？→ 完成
```

### discussion.md 格式

```markdown
## Round 1

### Leader 的修改
- 文件: src/utils.ts
- 改了什么: 把 forEach 改成 map，返回新数组
- 为什么改: Debug 用了 push 修改原数组，违反不可变原则

### Debug 的回复
- 判断: 同意
- 改了什么: 无额外修改
- 为什么: Leader 的改法确实更符合项目约定

## Round 2
...
```

### 设计考量

- **3 轮上限**：防止两个 AI 无限争论。人类也不会让两个同事永远讨论一个问题
- **文件记录**：discussion.md 让每轮讨论有据可查，跨轮次不丢上下文
- **上报机制**：到了上限，把完整讨论历史交给用户决策，而不是随便挑一个方案

## 异步调用模型

```
Leader                     Coder                    Debug
  │                          │                        │
  ├── 异步分派任务 ──────────→│                        │
  │   (run_in_background)    │                        │
  │                          │                        │
  │   继续做其他事...          │ 执行中...               │
  │                          │                        │
  │←── 系统通知完成 ──────────┤                        │
  │                          │                        │
  ├── 异步中转给 Debug ──────────────────────────────→│
  │   (run_in_background)    │                        │
  │                          │                        │
  │   继续做其他事...          │                        │ 审查中...
  │                          │                        │
  │←── 系统通知完成 ──────────────────────────────────┤
  │                                                   │
  ▼
```

好处：
- Leader 不阻塞，可以处理其他工作
- 未来可以通过增加 Coder/Debug 数量来水平扩展
- OpenClaw 自带消息队列，多个任务自动串行

## Leader 验收检查清单

Leader 不是随便看一眼就通过。验收时必须逐项检查：

- [ ] 代码实现了 tasks 中的所有任务
- [ ] 代码符合 specs 中的场景要求
- [ ] 测试通过（Coder 汇报的结果）
- [ ] 代码质量可接受（可读性、命名、无明显 bug）
- [ ] 遵循 AI-CONTEXT.md 中的项目约定

## 路由决策树

```
Leader 验收结果
    │
    ├── "这个功能没实现完" → 缺功能 → Coder + Debug
    │
    ├── "这里有 bug" → 有 bug → Debug
    │
    └── "都没问题" → 通过 → Tester
```

**不要搞混**：
- 缺功能 → Coder 写 + Debug 审查（需要新代码）
- 有 bug → Debug 直接修（已有代码有问题）

让 Debug 补功能或让 Coder 修 bug 都是错误的路由。
